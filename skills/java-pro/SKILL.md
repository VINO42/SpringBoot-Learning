## 核心能力

### 现代 Java 语言特性

- Java 21+ LTS 特性，包括虚拟线程（Project Loom）
- switch 表达式和 instanceof 的模式匹配
- Record 类作为不可变数据载体
- 文本块和字符串模板，提升代码可读性
- Sealed 类和接口，实现受控继承
- 使用 var 关键字的局部变量类型推断
- 增强的 switch 表达式和 yield 语句
- 外部函数和内存 API，实现原生互操作

### 虚拟线程与并发

- 虚拟线程：无需平台线程开销即可实现大规模并发
- 结构化并发模式：构建可靠的并发程序
- CompletableFuture 和虚拟线程的响应式编程
- 线程本地优化和作用域值
- 虚拟线程工作负载的性能调优
- 从平台线程迁移到虚拟线程的策略
- 并发集合和线程安全模式
- 无锁编程和原子操作

### Spring Framework 生态系统

- Spring Boot 4.x 及其 Java 25 优化特性
- Spring WebMVC 和 WebFlux 响应式编程
- Mybatis、Mybatis-flex、Mybatis-plus 性能特性

- Spring Cloud 微服务和分布式系统
- Spring Native 与 GraalVM 实现快速启动和低内存占用
- Actuator 端点用于生产监控和健康检查

### JVM 性能与优化

- GraalVM Native Image 编译用于云部署
- 针对不同工作负载模式（吞吐量 vs 延迟）的 JVM 调优

## 软件开发原则与定律

### 架构设计原则

| 原则          | 说明                             | 应用场景        |
|-------------|--------------------------------|-------------|
| **CAP 定理**  | 分布式系统最多只能同时满足一致性、可用性、分区容错性中的两项 | 分布式架构选型     |
| **康威定律**    | 系统设计会反映组织的沟通结构                 | 微服务拆分、团队组织  |
| **盖尔定律**    | 复杂系统必然是从简单系统演化而来，而非一开始就设计成复杂系统 | 系统演进策略      |
| **复杂性守恒定律** | 复杂性不会消失，只会转移                   | API 设计、框架设计 |
| **抽象泄漏定律**  | 所有抽象在一定程度上都会泄漏                 | 框架使用、底层调试   |
| **得墨忒耳定律**  | 最少知识原则，对象应该尽可能少地了解其他对象         | 代码解耦、封装     |

### 项目管理定律

| 原则            | 说明                        | 应用场景      |
|---------------|---------------------------|-----------|
| **布鲁克斯法则**    | 向进度落后的项目增加人手，只会使进度更加落后    | 项目规划、团队扩展 |
| **帕金森定理**     | 工作会自动膨胀，占满所有可用时间          | 迭代规划、时间管理 |
| **侯世达定律**     | 做事所花费的时间总是比你预期的要长         | 工期估算      |
| **90-9-1 法则** | 90%的用户只看不贡献，9%偶尔贡献，1%积极贡献 | 社区运营、开源项目 |

### 代码质量原则

| 原则         | 说明                | 应用场景      |
|------------|-------------------|-----------|
| **破窗效应**   | 代码质量问题会像破窗一样蔓延    | 代码审查、技术债务 |
| **奥卡姆剃刀**  | 如无必要，勿增实体         | 架构简化、需求评审 |
| **汉隆的剃刀**  | 能用愚蠢解释的，就不要归咎于恶意  | 团队协作、代码审查 |
| **过早优化效应** | 过早优化是万恶之源         | 性能优化时机    |
| **柯林汉定律**  | 调试代码的难度是最初编写代码的两倍 | 代码可读性     |

### 分布式系统原则

| 原则           | 说明                             | 应用场景      |
|--------------|--------------------------------|-----------|
| **分布式计算的谬论** | 网络可靠、延迟为零、带宽无限等假设都是错误的         | 分布式架构设计   |
| **隐式接口定律**   | 当接口有足够多用户时，你在契约中承诺的任何东西都成为隐式接口 | API 版本管理  |
| **鲁棒性原则**    | 发送时要保守，接收时要开放                  | 协议设计、接口对接 |

### 性能与扩展定律

| 原则         | 说明                       | 应用场景   |
|------------|--------------------------|--------|
| **阿姆达尔定律** | 并行计算的加速比受限于串行部分的比例       | 并发优化   |
| **摩尔定律**   | 集成电路上的晶体管数量每 18-24 个月翻一番 | 技术趋势判断 |
| **梅特卡夫定律** | 网络的价值与用户数的平方成正比          | 平台战略   |
| **里德定律**   | 社区网络的价值随规模呈指数增长          | 社区建设   |

### 认知与团队定律

| 原则                | 说明                      | 应用场景      |
|-------------------|-------------------------|-----------|
| **邓巴数字**          | 一个人能维持稳定社交关系的人数约为 150   | 团队规模控制    |
| **邓宁-克鲁格效应**      | 能力不足的人倾向于高估自己           | 技能评估、代码审查 |
| **彼得原理**          | 在等级制度中，每个人都会被提升到他不胜任的职位 | 团队管理      |
| **死海效应**          | 有能力的人更容易离开，留下的往往是能力较低的人 | 人才保留      |
| **帕累托法则 (80/20)** | 80%的效果来自 20%的原因         | 优化重点、缺陷分析 |

### 设计原则 SOLID

| 原则               | 说明                | 应用场景      |
|------------------|-------------------|-----------|
| **单一功能原则 (SRP)** | 一个类应该只有一个引起它变化的原因 | 类设计、模块划分  |
| **开闭原则 (OCP)**   | 对扩展开放，对修改关闭       | 框架设计、插件架构 |
| **里氏替换原则 (LSP)** | 子类必须能够替换其父类       | 继承设计、接口实现 |
| **接口隔离原则 (ISP)** | 不应该强迫依赖不使用的方法     | 接口拆分      |
| **依赖反转原则 (DIP)** | 依赖抽象而非具体实现        | 依赖注入、解耦   |

### 简洁原则

| 原则           | 说明                               | 应用场景      |
|--------------|----------------------------------|-----------|
| **DRY 原则**   | 不要重复自己 (Don't Repeat Yourself)   | 代码复用、抽象提取 |
| **KISS 原则**  | 保持简单愚蠢 (Keep It Simple, Stupid)  | 架构设计、代码实现 |
| **YAGNI 原则** | 你不需要它 (You Aren't Gonna Need It) | 功能裁剪、过度设计 |

### 其他重要原则

| 原则           | 说明                     | 应用场景         |
|--------------|------------------------|--------------|
| **墨菲定律**     | 可能出错的事终将出错             | 容错设计、异常处理    |
| **林纳斯定律**    | 给足够多的眼睛，所有 bug 都是浅显的   | 开源协作、代码审查    |
| **古德哈特定律**   | 当一个指标成为目标时，它就不再是好指标    | 绩效考核、质量度量    |
| **切斯特森围栏**   | 在拆除围栏之前，先了解它为什么存在      | 重构决策、遗留系统    |
| **Unix 哲学**  | 做一件事并做好；程序协同工作；处理文本流   | 工具设计、命令行程序   |
| **惠顿定律**     | 别做混蛋                   | 团队协作、社区交流    |
| **费茨法则**     | 目标越小或距离越远，命中所需时间越长     | UI 设计、交互优化   |
| **席克定律**     | 选项越多，决策时间越长            | UI 设计、API 设计 |
| **技术成熟度曲线**  | 技术从触发期到成熟期需要经历期望膨胀和失望期 | 技术选型         |
| **乔治·伯克斯定律** | 所有模型都是错的，但有些是有用的       | 架构建模、领域模型    |
